git结构

git命令行操作

本地库初始化

  • 初始化：git init

    在文件夹下面单击右键，选择git Bash here,再使用git init建造仓库。建立了一个名为.git的隐藏目录，可以用ls -1A查看

  • 用户命令级别

    • 项目级别、仓库级别：仅在当前本地库范围内有效(注意user.name和名字之间的空格)

      git config user.name my_name

      git config user.email email@123.com

      信息保存位置：.git里的config文件（使用cat .git/config查看）

    • 系统用户级别：在当前操作系统都有效

      git config --global user.name my_name

      git config --global user.email email@123.com

      信息保存位置：进入家目录（cd ~），里面有一个名为.gitconfig的隐藏目录（使用cat .gitconfig查看）

    • 两者都有时项目级别优先于系统用户级别。如果只有系统用户级别的签名，就以系统用户级别为准

添加提交以及查看工作状态

  • 新建txt文件：vim abc.txt

  • 查看工作状态：git status

    未在暂存区的文件为红色，暂存区内的文件为绿色

  • 提交至暂存区：git add abc.txt

  • 从暂存区删除：git rm --cached abc.txt

  • 从工作区删除：git rm abc.txt

  • 从暂存区提交到本地库：git commit -m "commit message" abc.txt

    • commit message:关于这次提交的备注

历史版本的前进和后退

  • 查看提交记录：

    git log，空格向下翻页，直到（END），b向上翻页，q退出。这是历史版本信息显示最完整的形式

    git log --pretty=oneline,每一条记录仅显示一行，更加简洁

    git log --oneline，在oneline基础上缩短了哈希值

    git reflog,相比oneline显示了指针到每个版本需要移动的次数

  • 前进/后退历史记录：

    • 基于索引值：

      git reset --hard 9a9ebe0 9a9abe0为局部索引值的示例，要根据历史记录来修改为具体的索引值

    • 使用^符号：只能后退

      git reset --hard HEAD^^^后退三个版本（每增加一个^,就多向后退一个版本）

    • 使用~符号：只能后退

      git reset --hard HEAD~3后退三个版本

    • reset命令的三个参数对比：

      可以使用git help reset查看reset文档

      • soft参数：仅仅在本地库移动指针

      • mixed参数：在本地库移动指针，同时重置暂存区

      • hard参数：在本地库移动指针，同时重置暂存区和工作区

  • 找回删除文件：

    • 前提：删除前，文件存在时的状态已经提交到了本地库

    • 操作：使用git reset --hard回退到之前的历史记录

      删除操作已经提交到本地库：指针指向历史记录

删除操作未提交到本地库：指针指向HEAD `git reset --hard HEAD`

  • 比较文件：

    git diff abc.txt没有参数时，工作区和暂存区的对应文件进行比较

    git diff HEAD^ abc.txt和上一个版本的对应文件进行比较

分支管理

  • 分支的好处：

    1. 同时推进多个功能开发，提高效率

    2. 如果一个功能开发失败，不会对其他分支有任何影响

  • 分支操作

    • 查看所有分支：git branch -v,绿色的是当前所在的分支

    • 创建新的分支：git branch another_branch

    • 切换分支:git checkout another_branch

    • 合并分支：git merge another_branch,注意要切换到接受新内容的分支上。合并完之后，被合并的分支依然存在

解决冲突

将特殊标识删除之后再人与人之间讨论如何合并两个文件，使用git add abc.txt将冲突标识为已解决

解决冲突后用git commit -m "commit message"提交到本地库，注意此时不可以带文件名

远程库操作

推送文件

  • 查看远程库地址别名：git remote -v

  • 添加别名：git remote add origin https://github.com/Yufei-Li/yufei-li.github.io.git，这样origin就可以代表一个github远程库，缩短了以后的命令长度

  • 推送：git push origin masterorigin为想推送的远程库，master为想要推送的分支

克隆文件

  • 命令：git clone https://github.com/Yufei-Li/yufei-li.github.io.git

  • 效果：

    1. 完整的把本地库下载到本地

    2. 创建origin远程地址别名

    3. 初始化本地库

远程库的拉取

  • fetch：git fetch origin masterorigin为远程地址库别名，master为远程分支名。将远程库的文件下载下来，但fetch并没有修改工作区的文件

    如果想要查看下载下来的内容，需要切换到git checkout origin/master，再使用cat abc.txt查看

  • merge：用fetch下载后，切换到mater分支下，使用git merge origin/master合并，工作区就可以显示远程库中的内容

  • pull：相当于fetch+merge，git pull origin master

协同开发时冲突解决

如果push到远程库时，不是基于远程库的最新版所做的修改，merge时候就会有冲突所以无法push。此时需要把最新版pull下来，进入merging状态，文件时会出现分支冲突时相同的特殊提示。之后就和分支冲突的解决方法一样，注意再次commit时不要带文件名
